BUILD THESE COMPONENTS IN ORDER:

===========================================================
COMPONENT 1 — PII DETECTOR
===========================================================

A function that scans any text string and finds:
- Student names (look them up from the real students table)
- Teacher names (look them up from the real teachers table)
- Email addresses (use regex)
- Phone numbers (use regex)
- Roll numbers or admission IDs (use regex + context)

Ask me for the exact table and column names before writing
any database query.

Cache the name list in memory with a 5 minute expiry so
you are not hitting the database on every AI call.

Return a list of everything found, with the exact position
in the string where each match starts and ends.

---

===========================================================
COMPONENT 2 — TOKENIZER
===========================================================

A function that takes the findings from the detector and
replaces each one with a short readable token like:
  STUDENT_A91F
  TEACHER_C22B
  EMAIL_F301
  PHONE_9A12

Generate the token using HMAC-SHA256 with a secret key
from environment variables so the same person always gets
the same token within a session.

Keep a temporary in-memory map of token to original value.
This map must never be written to the database, cache,
logs, or any file. It lives only for the duration of
one request and is discarded after.

Also write the reverse function that takes a response
from OpenAI and swaps the tokens back to real values.

---

===========================================================
COMPONENT 3 — OCR SANITIZER
===========================================================

A function specifically for answer sheet text extracted
by OCR before it is sent to OpenAI for evaluation.

Students often write their name, roll number, or class
at the top of their answer sheet. Strip all of that.

Remove patterns like:
  "My name is ..."
  "Roll No ..."
  "Class 10A ..."
  "I am ..."
  Any name found in the students table

Replace removed text with [STUDENT_IDENTIFIER].

Then run the full PII detector on the remaining text as
a second pass.

---

===========================================================
COMPONENT 4 — PRIVACY GUARD
===========================================================

A central orchestrator that all AI calls must pass through.

For a conversational query:
  1. Run PII detector on the user's prompt
  2. Replace PII with tokens
  3. Send the safe prompt to OpenAI
  4. Receive the response
  5. Check the user's role before unmasking
  6. Swap tokens back to real values
  7. Return the response
  8. Discard the token map

For an OCR evaluation:
  Same flow but run the OCR sanitizer in step 1 instead.

Role rules for unmasking:
  Student  ->  can only see their own name unmasked
  Teacher  ->  can see names of students in their classes
  Admin    ->  can see everything

If a role tries to unmask data it is not allowed to see,
throw an access denied error and return nothing.

===========================================================
COMPONENT 5 — CENTRAL LLM REGISTRY
===========================================================

This is the most important architectural rule in the system.

Create one central registry file that is the single source
of truth for everything related to LLM configuration.

No model name, temperature, token limit, or system prompt
should exist anywhere in the codebase except this file.
If any other file hardcodes these values, that is a bug.

This registry must define the following call types:
  QUERY       conversational analytics questions
  EVALUATE    answer sheet OCR evaluation
  INSIGHT     class-wide insight generation

For each call type define:
  - model name
  - temperature
  - max input tokens
  - max output tokens
  - timeout in seconds
  - the full system prompt text

Example system prompts to use as a starting point:

  QUERY system prompt:
  "You are an educational analytics assistant. Respond only
  with insights about academic performance and trends. Use
  only the identifiers given to you. Never invent or assume
  any identity."

  EVALUATE system prompt:
  "You are an exam evaluator. Evaluate the answer based on
  content quality only. The rubric will be provided with
  each request. Never reference student identity."

  INSIGHT system prompt:
  "You are a class performance analyst. Identify trends
  and areas needing attention from the data provided.
  Use only aggregated data and masked identifiers."

Ask me what model name and token limits to set before
writing the registry values.

===========================================================
COMPONENT 6 — AI GATEWAY
===========================================================

The ONLY file in the entire codebase allowed to import
or call the OpenAI library directly.

Every other part of the system calls this gateway.
The gateway reads ALL config from the Central LLM Registry.
The gateway calls the Privacy Guard before every request.

Responsibilities:

  - Read model, temperature, max tokens from the registry
    for the specific call type being made
  - Read system prompt from the registry for that call type
  - Before sending: run one final regex sweep on the masked
    prompt to catch anything the detector missed
    If anything suspicious is found, block the request
    and throw a named error called UNSAFE_PAYLOAD
  - Enforce max input token limit from the registry
  - Make the OpenAI API call using registry config
  - Write a safe audit log after every call containing
    ONLY these fields:
      request_id, timestamp, call_type, model,
      token_usage, latency_ms
    NEVER log the prompt, response, OCR text, or token map
  - If OpenAI returns an error, wrap it in a generic
    message before returning to the caller — never expose
    raw OpenAI error messages to the client

Expose only these public functions:
  askQuestion(prompt, userContext)
  evaluateAnswerSheet(ocrText, rubric, userContext)
  generateClassInsight(analyticsData, userContext)

No route or service anywhere may call OpenAI any other way.
